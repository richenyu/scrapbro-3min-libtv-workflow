# LibTV 提示词逻辑优化工作流 V1

用途：把现有“因果连续性工作流”改造成专门审查、优化、输出 LibTV / 里布 15 秒视频提示词的工作流。

适用场景：
- 剧本改成 LibTV 15 秒提示词
- 已有提示词做逻辑审计
- 动作戏 / Boss 战 / 追逐 / 逃亡提示词优化
- 修正 AI 视频生成失败问题
- 提取某一 Part 的里布发布版提示词

---

## 一、原逻辑工作流是什么

原工作流叫：

`Causal Continuity Logic Review Workflow v2.0`

它的核心不是“润色”，而是检查每一个事件是否有：

```text
原因 -> 限制 -> 选择 -> 代价 -> 结果
```

也就是说，一个剧情不能只是“他去了、他打了、他赢了”。必须回答：

```text
为什么现在必须做？
角色为什么会做？
做的时候受什么限制？
他付出了什么代价？
做完以后改变了什么？
下一场为什么自然发生？
```

---

## 二、原工作流完整流程

### 1. 建输入包

先收集这些信息：

```text
待审文本
上一集 / 上一场结尾
人物位置、伤势、情绪、关系、已知信息
装备、工具、道具、钱、资源状态
敌人知道什么、得到了什么、下一步会做什么
伏笔、未解钩子、下一集入口
```

如果这些信息不清楚，就不能直接审剧情，要先补状态表。

### 2. 分层审查

把文本拆成不同层级：

```text
系列主线
阶段剧情
单集剧情
单场戏
动作节拍
```

比如赛铁坡这一集：

```text
系列主线：黑盒、旧世界、能源系统、怪物生态
单集目标：拿到完整能源锁环
单场戏：外路遭遇、小怪围攻、毒女出现、Boss 战、黑龙钩子
动作节拍：摩托甩尾、断链卡轮、酸泥急刹、Boss 冲撞、取核封存
```

### 3. 过三道门

第一道门：大因果门

检查：

```text
动机是否成立
触发事件是否清楚
线索是否可见
角色选择是否符合性格
能力是否有限制
有没有代价
结果是否改变局面
敌人有没有获得信息或制造后果
下一场是否自然接上
```

第二道门：小逻辑门

检查：

```text
时间是否够
空间位置是否清楚
人物从哪里到哪里
手里拿着什么
装备是否还能用
伤势是否延续
敌人为什么这样反应
道具从哪里来
动作打在哪里
打完落点在哪里
```

第三道门：戏剧波峰门

检查：

```text
有没有压力
有没有失误或受挫
有没有反转
有没有小胜
有没有代价
有没有下一段钩子
```

### 4. 给问题分级

```text
S 级：结构断裂，主线崩，动作不可能，必须重写
A 级：重要因果缺失、状态缺失、地理不清，必须修
B 级：局部动作、情绪、道具、代价不够清楚，需要补
C 级：表达可以更清楚，逻辑基本成立
```

### 5. 按顺序修

修复顺序固定：

```text
先修大因果
再修状态连续
再修动作地理
再加强戏剧波峰
最后删掉多余解释
```

不能先加炫酷动作。逻辑不通的时候，加动作只会更乱。

### 6. 复审

改完以后必须再审一遍：

```text
是否通过大因果门
是否通过小逻辑门
是否通过戏剧波峰门
是否还能被 AI 视频拍出来
```

如果复审不过，继续修。

## 三、改造成 LibTV 提示词专用以后，要增加什么

原工作流主要审“剧情逻辑”。但 LibTV 提示词还多了几个特殊问题：

```text
15 秒能不能拍完
图片引用是否正确
上传素材顺序是否合理
角色锁定图会不会污染
动作地理是否足够给 AI 看懂
负面词是否太多
是否容易生成字幕、标题、水印
对白会不会太密
上一段和下一段是否能接
```

所以 LibTV 专用版要在原三道门上，加六个专门检查门。新增的“细节动作因果门”专门检查打击、顶飞、摔落、砍中、撞击等动作是否真的按物理因果发生，避免 AI 把近身动作生成成远程射线、滑倒、简单对撞或不清楚的混乱特效。

## 四、LibTV 专用工作流 V1

### 总流程

```text
读取剧本 / 提示词
-> 建 LibTV 输入包
-> 建状态连续表
-> 建素材引用表
-> 过剧情因果门
-> 过 15 秒可拍门
-> 过动作地理门
-> 过细节动作因果门
-> 过素材锁图门
-> 过对白声音门
-> 过平台防错门
-> 分级标注问题
-> 按顺序修复
-> 输出发布版提示词
-> 复审
-> 生成审计报告
```

## 五、LibTV 输入包

每次优化提示词，先收集：

```text
1. 原剧本
2. 当前提示词版本
3. 要生成的是第几段，时长几秒
4. 上一段最后一帧是什么
5. 本段第一帧是什么
6. 本段最后一帧是什么
7. 下一段第一帧怎么接
8. 角色锁定图有哪些
9. 怪兽 / Boss 锁定图有哪些
10. 场景锁定图有哪些
11. 道具 / HUD 锁定图有哪些
12. 本段真正要解决的一个主动作目标是什么
```

如果是单独提取某一段发布版，还要加：

```text
视频名称
建议参数
素材上传顺序
提示词正文
```

## 六、LibTV 状态连续表

每一段都要记录：

```text
人物位置
人物伤势
装备状态
道具在哪里
敌人在哪里
上一段留下的危险
本段结束后的新状态
下一段承接点
```

例如赛铁坡：

```text
Part01-03：后箱只有裂开的半环样片
Part04：右手套被腐蚀出细裂口
Part06：防腐塞临时封住手套裂口
Part08：防腐塞开始冒烟，稳定时间变短
Part10：防腐塞裂成两半
Part11-12：右手套持续冒细烟
```

状态不能自动恢复。

---

## 七、LibTV 素材引用表

每个 `@` 都要回答：

```text
这个图本段是否真的出现？
它是人物、场景、怪兽、道具还是 HUD？
它排第几张上传？
它是主锁定图还是补充参考？
会不会把不该出现的东西带进画面？
```

规则：

```text
本段不出现的角色不要上传
Boss 没正式登场前不要放 Boss 全身锁图
完整道具没出现前，不要让 AI 生成完整道具
使用完整道具参考图时，要写清本段只参考材质 / 形状 / 裂纹
最多 9 张；前置视频节点不占图片位；不再上传上一段最后一帧 / 尾帧承接图 / 伪真尾帧图
素材优先级从场景、主角、伙伴、载具、当前怪物、当前道具、分镜参考图开始
```

## 八、八道检查门

### 1. 剧情因果门

问：

```text
本段为什么发生？
上一段怎么推到这一段？
主角为什么这样选择？
敌人为什么这样攻击？
本段结果如何推动下一段？
```

不通过表现：

```text
怪物突然出现
Boss 像刷新出来
道具像凭空出现
主角突然知道某个信息
任务物提前出现或消失
```

### 2. 15 秒可拍门

问：

```text
15 秒里是否只有一个主动作目标？
对白是否过密？
动作、表情、道具特写是否塞太多？
是否可以拆成 4-6 个小镜头？
最后一帧是否能自然接下一段？
```

原则：

```text
15 秒优先保动作和状态，少解释
一段只解决一个主问题
对白宁可短，不要压过动作
```

### 3. 动作地理门

问：

```text
谁在前方？
谁在中景？
谁在后方？
敌人从哪个方向冲出？
攻击目标是谁？
主角用哪只手 / 脚 / 装备反击？
命中点在哪里？
打完落点在哪里？
```

不通过表现：

```text
角色瞬移
怪物攻击摄影机
追击者跑到主角前方
RAM 位置忽左忽右
Boss 和摩托空间关系混乱
```

### 4. 细节动作因果门

问：

```text
这个动作的出发点在哪里？
角色用哪个身体部位、武器或装备发力？
发力前有没有靠近、助跑、蓄力、转身、甩尾或换重心？
真正接触点在哪里，是肩颈、侧腹、前轮、车门、地面裂缝，还是敌人的某个护甲缝？
受力方向是什么，敌人是横向飞、向后飞、向上离地、翻滚，还是只是倒下？
落点在哪里，是废车门、排水沟、路肩、车底、墙面，还是地面？
声音是否跟接触点一致，是闷响、金属撞击、骨甲碎裂、冷电爆裂、落地翻滚，还是泛泛的爆炸声？
动作结果是否推动下一镜头，而不是只制造一团特效？
```

近身打击硬规则：

```text
拳头、拳套、电拳、冰伏拳套、脚踢、肩撞、头槌、刀砍，默认都是近身接触动作。
如果剧情要“打飞敌人”，必须明确写：靠近方式 -> 发力部位 -> 接触点 -> 受力方向 -> 飞出轨迹 -> 落点。
不要只写“电拳一击”或“冰伏爆开”，否则 AI 容易生成远程光束、剑气、射线、电流发射或让敌人自己滑倒。
```

伙伴顶飞硬规则：

```text
伙伴把敌人顶飞时，必须写清：助跑角度、低头或压肩动作、顶撞部位、敌人被顶离地、翻滚方向、落点。
不要只写“撞偏”“卡位”“对撞”，否则 AI 容易生成双方简单撞在一起，缺少顶飞结果。
```

不通过表现：

```text
本来是拳头打飞，画面却像剑气、射线、远程电流或枪械射击。
本来是主角打飞敌人，画面却变成敌人自己滑倒。
本来是 RAM 顶飞敌人，画面却变成 RAM 和敌人正面对撞后都停住。
接触点被烟尘、强光、运动模糊挡住，看不出谁打中了谁。
敌人没有明确飞出轨迹和落点，只是画面一团乱。
动作声音和接触点不匹配，例如明明是肩撞却只有泛泛爆炸声。
```

输出时必须写：

```text
动作起点
发力部位
接触点
受力方向
飞出 / 倒下 / 撞击轨迹
落点
对应声音
禁止误生成项
```

案例沉淀：

```text
赛铁坡 Part01 第 7 格：ScrapBro 必须用右手冰伏电拳近身接触打中小野猪肩颈和侧脸，把它横向打飞撞进废车门；禁止生成剑气、光束、射线、远程电流或小野猪自己滑倒。
赛铁坡 Part01 第 8 格：RAM 必须从右侧斜切到左前方，低头助跑，用肩颈和胸前装甲顶中小野猪侧腹，把它顶离地、横向翻滚进排水沟；禁止生成双方简单对撞。
```

### 5. 素材锁图门

问：

```text
@标签和上传素材是否一一对应？
素材是否会污染当前段？
图越多是否反而让 AI 混淆？
本段最关键的场景锁定图是否排在前面？
人物锁定图是否会导致战斗中正脸看镜头？
```

输出时必须写：

```text
素材上传顺序
每张图用途
本段提示词内的 @引用
```

### 6. 对白声音门

问：

```text
谁在说话？
说话时角色在做什么？
声音从现场来，还是像旁白？
是否出现中文配音风险？
是否会生成字幕？
```

规则：

```text
英文对白可以保留
中文翻译只供理解
不生成字幕
不要旁白
不要口播感
角色不能正对镜头说台词
```

### 7. 平台防错门

检查：

```text
不要现代停车场
不要现代车流
不要字幕
不要标题
不要水印
不要动画风
不要游戏 CG 感
不要角色复制
不要残影分身
不要强运动模糊遮动作
```

负面约束使用原则：

```text
主提示词里优先写正向动作
防错区再集中写禁止项
不要在每一句动作里反复塞否定词
```

### 8. 复审门

修完后检查：

```text
剧情是否更清楚？
动作是否更可拍？
15 秒是否能完成？
素材引用是否更稳？
是否删掉了多余解释？
是否生成了可直接复制的发布版？
```

## 九、问题分级：LibTV 专用版

S 级：

```text
主因果断裂
角色状态前后矛盾
Boss / 怪物行为不可能
任务物提前出现或丢失
动作空间完全拍不清
15 秒塞进两个以上大事件
```

A 级：

```text
道具来源不明
怪物出现原因不明
上一段和下一段接不上
重要装备状态没有延续
上传素材会导致明显错图
```

B 级：

```text
对白偏密
动作接触点不清
落点不清
动作发力方式不清，容易把近身打击生成远程射线
敌人被打飞或顶飞的受力方向不清
伙伴顶飞动作写成“撞偏 / 卡位 / 对撞”，缺少离地、翻滚和落点
声音锁定不够
场景锁定不够
负面约束写得太散
```

C 级：

```text
表达可更短
视频名称可更清楚
上传顺序可更顺
单句可更像提示词
```

## 十、修复顺序：LibTV 专用版

固定顺序：

```text
1. 修主因果
2. 修状态连续
3. 修素材引用
4. 修动作地理
5. 修细节动作因果
6. 修 15 秒密度
7. 修声音对白
8. 修平台防错
9. 输出发布版
10. 复审
```

不要一上来就改文采。

## 十二、赛铁坡案例里的工作流应用

原问题：

```text
@Energy_Lock_Ring 可能让 AI 提前生成完整锁环
Boss 出场低频脉冲传导原因不够可视化
断链来源不够清楚
毒核心像游戏掉落物
黑龙钩子震动对象不够清楚
```

工作流修法：

```text
样片和完整锁环分开
补低频脉冲沿黑绿泥痕和腥盐砖传导
断链改成来自炮座旁废车堆
毒核心改成扫描、拨管、夹出、封存
黑龙钩子改成完整锁环、半环样片、毒核心封存盒同时反应
```

结论：

```text
V3 通过逻辑审计，可作为当前 LibTV 赛铁坡测试版。
```
